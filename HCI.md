# 智能体安全协同与调用 (agent-oncall) 用户交互体验设计说明书 (HCI.md) 📞

本文件从**用户交互与人机协同 (Human-Computer Interaction)** 的视角，总结智能体使用 `agent-oncall` 层进行安全连接、授权与协同调用时的全部交互体验。

---

## 🏗️ 1. 身份与名片交换体验 (Identity & Contacts Exchange)

智能体在网络中的身份标识为 **URN**（例如 `urn:hermes:agent:alice`）。在两个智能体能够开始安全协作之前，用户需要进行身份和信任关系的初始化：

- **生成名片**：
  - 用户在终端输入：`./agent-comm share`
  - 智能体输出本机的安全连接名片（包含 URN、物理网络地址以及静态加密公钥的文本块）。
- **导入名片**：
  - 用户获取对端智能体的名片文本块，通过 `./agent-comm import <名片>` 导入。
  - 智能体导入成功后，会将对端 URN 登记到联系人列表中，并在后台建立双棘轮加密会话。
  
> [!NOTE]
> 从用户的视角来看，名片交换是一次性的配对行为（类似于添加社交好友或蓝牙配对），一旦配对成功，后续通信将被静默端到端加密保护。

---

## 👥 2. 动态信任级别设定 (Dynamic Trust Tiers)

用户可以为不同的智能体分配不同的信任级别，不同的信任级别将直接决定其对本地工具和意图（Intents）的访问权限：

- **配对默认状态**：新导入的智能体默认标记为 `Tier_3_Stranger` (陌生人)。
- **升级信任级别**：
  - 用户认为对端智能体安全时，可以运行：`./agent-comm trust <URN>`，将其升级为 `Tier_2_Friend` (好友) 或 `Tier_1_Family` (亲人)。
  - 用户也可以运行：`./agent-comm untrust <URN>` 随时撤销信任。
- **用户直观感受**：
  - 信任级别为“陌生人”的智能体只能发现我们公开的最基础工具（如 `info.hello`）。
  - 信任级别为“好友”的智能体可以查询日程安排等。
  - 信任级别为“亲人”的智能体可以执行更敏感的操作。

## 🚨 3. 人机确认拦截体验 (Human-in-the-Loop - HITL)

当远程智能体尝试调用本机的**高风险意图**（如订票、支付、删除文件、清空日程）时，`agent-oncall` 会启动安全拦截并向本地宿主用户弹出确认窗口。

### 终端下的交互体验
智能体会在屏幕上打印醒目的警告，并等待用户手动确认（输入 `y` 或 `n`）：

```text
📞 ================= [HITL INTERCEPT] =================
👤 Caller:    urn:hermes:agent:bob
🔧 Intent:    calendar.book_event
📝 Arguments: {"event_title": "Urgent Meeting with CEO", "time": "14:00"}
=========================================================
Do you approve this operation? (y/n): 
```

### 体验设计要点
1. **意图透明**：展示请求者身份（Caller）、准备执行的意图（Intent）以及具体入参（Arguments），确保用户在完整知情的前提下做出决策。
2. **默认拒绝**：若用户直接按下回车、或发生中断 (Ctrl+C/EOF)，系统将默认选择拒绝（Safety-by-Default），确保无人值守时的绝对安全。
3. **异步/可插拔**：本库提供 `HITLHandler` 接口，宿主框架可轻松将其重定向至 UI 弹窗（例如 App 里的 Push 确认按钮）或 IM 对话。

## 🎟️ 4. 权限临时委托体验 (Capability Tokens - HCT)

在多智能体协同工作中，用户可能需要临时授权某个“陌生人”智能体访问特定资源（如允许工作智能体临时写入日程）。`agent-oncall` 提供了**基于能力的授权令牌 (HCT)**。

- **签发 HCT**：
  - 用户（作为授权人）命令自己的智能体签发一个令牌，将 `calendar:write` 权限授权给对端的陌生人 URN，并设置有效期（如 60 秒）和限制条件。
- **携带令牌调用**：
  - 远程陌生人智能体再次发起调用时，在其调用请求中附带该 HCT 令牌。
  - 本地智能体的策略引擎在发现对方虽然是“陌生人”，但携带了有效的 Alice 签名 HCT，便会予以临时通行。
- **用户体验**：
  - 用户无需永久修改对等节点的信任级别，仅通过发放“临时门禁卡”即可完成单次或短期工作委托，到期自动失效。

---

## 🤖 5. 静默意图对齐体验 (Service Description Alignment - SDA)

在服务发现过程中，对端暴露的工具描述往往具有模糊性（例如我们的参数叫 `date`，对方的字段叫 `time_str`）。

- **无感知的后台对齐**：
  - 从用户视角看，智能体在后台静默发起对话。
  - Requester 智能体自动理解对端描述，尝试生成测试请求发送给对端，并核对返回结果是否符合预期。
  - 对齐成功后，Requester 会在本地建立映射缓存，此后便可以直接可靠地调用该服务。
- **体验优势**：
  - 避免了由于厂商不同、字段命名差异导致的协同失败，也无需用户介入手动做 API 参数适配。

---

## 💾 6. 数据驱动的可视化配置数据库 (Data-Driven Configuration DB)

为了方便用户直接或通过智能体助手轻松管理信任关系与权限设定，所有的策略和联系人信息都保存在一个标准的 **JSON 配置文件**（如 `trust_db.json`）中。

- **易于阅读与修改**：
  - 信任数据库结构清晰（包含全局 `tier_permissions` 信任级别权限集，以及 `contacts` 联系人卡片信息与个性化 `allowed_intents` 授权过滤列表）。
  - 用户或智能体助手可以直接对该文件进行 JSON 读写与修改，无需调用复杂的 API 接口即可完成运行时权限重载。
- **防止版本冲突**：
  - 本地运行生成的 `trust_db.json` 文件已被默认加入项目的 `.gitignore` 中。这意味着当用户通过 `git pull` 更新项目代码时，本地的策略数据库**永远不会被覆盖或丢失**，保证了策略配置的持久与安全。
